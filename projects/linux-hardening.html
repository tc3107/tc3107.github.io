Case Study

Security Operations & Hardening

Practical security work across a small business environment: reducing exposure, protecting access, monitoring for problems, keeping systems recoverable, and making security usable for non-technical employees.

Goal

Maintain a secure and reliable baseline for Linux servers, connected PCs, storage services, remote access, cloud systems, and business data without disrupting daily operations.

Security Areas Covered

  • Access control: user permissions, account access, SSH administration, and least-privilege thinking.
  • Network exposure: firewalling, allowed services, DNS/domain considerations, and remote access boundaries.
  • System hygiene: updates, service review, log checks, disk usage monitoring, and configuration awareness.
  • Backup resilience: keeping important business data recoverable through local and cloud-backed backup workflows.
  • Operational monitoring: uptime checks, alerts, and custom scripts/software where existing tooling was not enough.

Implementation Approach

  • Reviewed exposed services and firewall behaviour before and after changes.
  • Used Linux logs and service status checks to investigate operational and access-related issues.
  • Maintained update and restart awareness so security work did not accidentally interrupt business workflows.
  • Applied practical hardening measures around SSH, permissions, backups, and network access.
  • Designed monitoring and alerting around real operational risks such as service failure, storage issues, and availability problems.
  • Kept sensitive details out of public documentation while documenting concepts, workflows, and decision-making.

Cybersecurity Relevance

This work maps well to junior SOC, NOC, support, and security analyst roles because it involves triage, logs, access issues, suspicious behaviour awareness, reliability, data protection, and clear documentation. The environment is small, but the habits are the same ones larger teams depend on.

What I Learned

  • Good security is operational: it has to be monitored, documented, and recoverable.
  • Access control and backups are not side tasks; they are core security controls.
  • Non-technical users need respectful communication, not blame, when security affects workflow.
  • Small environments teach broad responsibility because there is no separate networking, security, support, and infrastructure team to hand work to.
